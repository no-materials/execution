# Verifier: add lint mode (non-fatal warnings)

## Design

Add an optional “lint” mode to verification that produces *non-fatal* warnings about
low-quality (but still valid) bytecode. This keeps verification crisp (pass/fail)
while still giving frontend authors actionable feedback.

API shape:
- Keep existing `verify_program*` behavior unchanged.
- Add sibling entrypoints that return `Vec<VerifyLint>` alongside success:
  - `verify_program_with_lints(&Program, &VerifyConfig) -> Result<Vec<VerifyLint>, VerifyError>`
  - `verify_program_owned_with_lints(Program, &VerifyConfig) -> Result<(VerifiedProgram, Vec<VerifyLint>), VerifyError>`

Initial lints (cheap, high-signal):
- `UnreachableBlock`
- `DeadStore`
- `RedundantMove`
- `UnusedCallReturn` (call/host_call)

Implementation notes:
- Best-effort: lint collection must not affect acceptance; errors remain errors.
- Keep `no_std + alloc`; no new dependencies.
- Prefer deriving lints from existing verifier metadata (CFG + per-instr reads/writes) so
  costs are modest and the mapping is correct.

## Acceptance

- New verifier APIs exist and are documented.
- Lints are non-fatal (existing verification acceptance unchanged).
- At least one deterministic unit test covers each lint kind.
- `cargo fmt`, `cargo clippy -- -D warnings`, and `cargo test --workspace --all-features` pass.
